2. 分布式系统基础概念

定义

分布式系统是若干独立自主计算机的集合，这些计算机对于用户来说像是单个耦合系统。

著名定义（Leslie Lamport）：
“A distributed system is one in which the failure of a computer you didn’t even know existed can render your own computer unusable.”

特性

  1. 自主性

    • 计算节点硬件或软件进程是独立的
    • 物理分布，逻辑集中

  2. 耦合性

    • 用户或应用程序感觉系统是一个整体
    • 节点之间需要相互协作
    • 个体独立，整体统一

  3. 其他特性

    • 构成组件被所有用户共享
    • 系统资源可能不允许访问
    • 软件运行在不同处理器上的多个并发进程上
    • 允许多点控制
    • 允许多点失效

3. 分布式系统目标

主要目标包括四个方面：

  1. 资源访问

    • 让用户方便地访问资源

  2. 透明性

    • 隐藏资源在网络上的分布

  3. 开放性

    • 访问接口的标准化

  4. 可扩展性

    • 系统在规模、地域、管理上的可扩展性

透明性详解

透明性类型包括：

透明性类型 说明
访问 隐藏数据表示形式的不同以及资源访问方式的不同
位置 隐藏资源所在位置
迁移 异常资源是否移动到另一个位置
重定位 隐藏资源是否在使用过程中移动到另一个位置
复制 隐藏是否对资源进行复制
并发 隐藏资源是否由相互竞争的用户共享
故障 隐藏资源的故障和恢复
持久化 隐藏数据在主存和磁盘这一事实

注意：完全透明性是不可取也是难以实现的，原因包括：

  • 可能掩盖通信的性能问题
  • 完全隐藏网络和节点的失效是不可能的
  • 完全的透明性可能牺牲性能
  • 保证复制节点与主节点的一致性需要时间

4. 可扩展性

三个方面的可扩展性

  1. 规模可扩展性：用户数量和进程数量增加
  2. 地理可扩展性：节点之间的最大物理位置
  3. 管理可扩展性：管理域的数量

扩展技术

  1. 隐藏通信延迟

    • 利用异步通信技术
    • 设计分离的响应消息处理器
    • 将计算从服务器端移动到客户端

  2. 分布

    • 在多个机器上划分数据和计算
    • 分布式的命名服务器（DNS）
    • 分布式信息系统（WWW）

  3. 副本

    • 在多个不同机器上创建多个数据副本
    • 复制文件服务器和数据库
    • Web站点镜像
    • Web缓存
    • 文件缓存

5. 分布式系统类型

1. 集群计算系统

  • 通过LAN连接的高端计算系统
  • 同构：相同的OS，近乎相似的硬件
  • 单个管理节点

2. 网格计算系统

  • 由各地节点构成的系统
  • 异构
  • 包含多个组织
  • 易扩展到广域网环境

3. 云计算

分为四层：

  1. 硬件层：处理器、路由、电源和冷却系统
  2. 基础设施层：部署虚拟化技术
  3. 运行平台层：提供高层抽象
  4. 应用程序层：实际的应用程序

4. 普适计算系统特点

  • 分布式：设备通过网络连接
  • 交互：用户和设备之间交互高度隐蔽
  • 上下文可感知：系统知晓用户上下文
  • 自主性：设备自主运行
  • 智能：系统可处理动态行为和交互

5. 移动计算系统

特征：

  • 大量不同的移动设备
  • 设备位置随时间变化
  • 设备间通信困难

6. 传感网络

特点：

  • 节点数量众多（10~1000）
  • 简单（内存小、计算能力低、通信效率低）
  • 通常电池驱动

6. 分布式系统面临的挑战

  1. 系统设计

    • 正确的接口设计和抽象
    • 如何划分功能和可扩展性

  2. 一致性

    • 如何一致性共享数据

  3. 容错

    • 如何保障系统在出现失效情况下正常运行

  4. 不同的部署场景

    • 集群
    • 广域分布
    • 传感网络

  5. 实现

    • 如何最大化并行
    • 性能瓶颈是什么
    • 如何平衡负载

2. 软件体系结构

2.1 基本概念

软件体系结构的”样式”包含以下几个方面：

  1. 可替换性：组件具有定义良好的接口
  2. 组件之间互联的方式
  3. 组件之间交换的数据
  4. 组件以及连接器的协同配置方式

2.2 软件体系结构风格分类

  1. 分层体系结构

    • 主要用于客户端-服务器模型

  2. 基于对象的体系结构

    • 主要用于分布式对象系统

  3. 基于事件的结构

    • 如发布/订阅(Pub/Sub)结构

  4. 共享数据空间结构

    • 进程的空间(“匿名化”)和时间(“异步化”)解耦

3. 系统体系结构

3.1 类型划分

  1. 集中式体系结构(master-slave)

    • 整个系统包含一个控制中心
    • 协同系统的运行

  2. 非集中式组织结构(P2P)

    • 系统没有整体控制中心
    • 各个节点独立自主运行

  3. 混合组织结构

    • 同时包含集中式和非集中式结构
    • 如点对点文件共享系统

3.2 客户端-服务器模型

特征：

  1. Server进程

    • 实现特定服务的进程

  2. Client进程

    • 通过发送请求获取服务
    • 等待服务器回复

  3. 通信协议

    • HTTP/HTTPS: REST API (GET、POST等)
    • AJAX: 异步请求
    • RPC: XMLRPC、SOAP、gRPC

3.3 应用分层

传统三层视图：

  1. 用户接口层

    • 系统与用户直接交互的单元
    • 如显示管理等

  2. 应用处理层

    • 包含应用的主要函数
    • 不与具体数据绑定

  3. 数据层

    • 管理应用使用的实际数据

4. RESTful架构

4.1 基本概念

  • REST (Representational State Transfer)
  • 将分布式系统看做资源的集合
  • 由组件独立管理

4.2 特点

  1. 资源可以通过命名机制标识
  2. 所有服务提供相同的接口
  3. 消息是完全自描述的
  4. 执行完操作后不记录调用者信息

4.3 案例：Amazon S3

  • 对象(objects)放在桶(Buckets)中
  • 桶不能嵌套
  • 通过HTTP协议进行操作
  • 典型操作：
    • 创建桶或对象：PUT操作
    • 列出对象：GET操作
    • 读对象：通过URI执行GET

5. 非集中式体系结构

5.1 分布方式

  1. 垂直分布

    • 系统逻辑分层
    • 不同层次分布在不同机器上

  2. 水平分布

    • 客户或服务器分成逻辑相等的几部分
    • 每部分相对独立
    • 分布在不同机器上

  3. 点对点系统

    • 水平分布
    • 进程完全相同
    • 既是客户端又是服务器
    • 无中心化系统

5.2 P2P系统类型

  1. 结构化P2P系统

    • 节点按特定分布式数据结构组织
    • 主要实现：
      • Chord结构(基于DHT)
      • CAN(Content Addressable Network)

  2. 无结构化P2P系统

    • 节点的邻居随机选择
    • 使用泛洪或随机游走方式进行数据搜索

  3. 混合P2P系统

    • 部分节点具有良好组织方式
    • 如超级对等节点设计

5.3 超级对等节点

功能：

  • 维护索引
  • 监控网络状态
  • 建立连接
  • 提高搜索性能
  • 协助数据放置决策

6. 分布式系统的自我管理

6.1 自主计算特性

  1. 自我管理
  2. 自我恢复
  3. 自我配置
  4. 自我优化

6.2 实现机制

  • 通过反馈控制循环实现自适应性
  • 使用监控系统(如Prometheus + Grafana)实现可观测性

6.3 中间件支持

  1. 拦截器(Interceptors)
  2. 自适应中间件特点：
    • 功能与非功能部分分离
    • 支持计算反射
    • 基于组件的设计

分布式系统中的进程与虚拟化

1. 分布式系统中的线程基础

1.1 基本概念

  • 处理器: 提供和运行一系列指令集合的硬件平台
  • 线程: 最小的可执行一系列指令的软件处理器
  • 进程: 包含多个线程的软件处理单元，线程需要在进程的上下文中执行

1.2 上下文类型

  1. 处理器上下文

    • 处理器运行指令所需的最小寄存器数据集
    • 包括：栈指针、地址寄存器、程序计数器等

  2. 线程上下文

    • 执行指令所需的最小寄存器和内存数据集
    • 包括：处理器上下文、状态等

  3. 进程上下文

    • 执行线程所需的最小寄存器和内存数据集
    • 包括：线程上下文、MMU寄存器值、TLB等

1.3 使用线程的原因

  1. 避免不必要的阻塞

    • 单线程进程在I/O操作时会被阻塞
    • 多线程进程可在I/O时切换到其他线程

  2. 更好地发挥并行性

    • 多线程进程可在多核或多处理器CPU上并行执行

  3. 避免进程上下文切换

    • 使用多线程而非多进程来构建大型应用
    • 线程切换代价远小于进程切换

2. 线程的实现

2.1 实现方式

  1. 用户级线程

    • 优点：上下文切换代价小
    • 缺点：线程阻塞会导致整个进程阻塞
    • 示例：libthread, goroutine

  2. 内核级线程

    • 优点：线程阻塞不影响其他线程
    • 缺点：每个线程操作都需要系统调用

  3. 混合实现：轻量级进程(LWP)

    • 结合用户级和内核级线程的优势
    • 用户线程通过LWP执行系统调用

2.2 LWP主要操作

  1. 用户级线程执行系统调用时绑定到LWP
  2. 内核调度负责调度另一个LWP
  3. 线程阻塞时可切换到同一LWP的其他线程
  4. 空闲LWP可能被内核回收

3. 虚拟化(Virtualization)

3.1 虚拟化的重要性

  1. 硬件更新快于软件
  2. 需要灵活的可移植性和代码迁移
  3. 避免失效和攻击隔离
  4. 屏蔽底层软硬件异构性

3.2 接口层次

  1. 指令集架构

    • 特权指令：仅操作系统可执行
    • 通用指令：任何程序可执行

  2. 系统调用

    • 操作系统提供的函数

  3. 库函数调用(API)

3.3 虚拟化类型

  1. 进程虚拟机(Process VM)

    • 运行在OS之上的解释器或模拟器
    • 例如：JVM, Qemu

  2. 原生虚拟机监控器(Native VMM)

    • 直接运行在硬件上
    • 包含最小的操作系统

  3. 主机虚拟机监控器(Hosted VMM)

    • 需要完整的操作系统
    • 运行在现有OS之上

4. Docker容器技术

4.1 基本概念

  • 镜像(Image): 标准化的软件包，包含应用及依赖
  • 容器(Container): 应用运行的独立环境
  • 仓库(Registry): 存储和分发镜像的服务

4.2 Docker vs 虚拟机

特点比较：

  • Docker共享主机OS内核
  • 容器启动更快
  • 资源利用更高效
  • 隔离级别相对较低

5. 代码迁移

5.1 迁移目的

  1. 负载分布

    • 确保服务器资源充分利用
    • 最小化通信代价

  2. 灵活性

    • 按需将代码迁移到客户端

5.2 迁移类型

  1. 弱移动性

    • 仅迁移代码和数据片段
    • 需要重启执行
    • 包括代码推送和代码拉取

  2. 强移动性

    • 迁移所有组件，包括执行状态
    • 支持迁移和克隆操作

5.3 虚拟机迁移方案

  1. 推送内存页面并更新修改
  2. 停机迁移后重启
  3. 按需拉取内存页面

注意：完整虚拟机迁移可能需要几十秒，服务可能短暂不可用。

分布式系统通信

1. 基础网络模型

1.1 网络层次结构

  1. 物理层

    • 负责数据的实际传输
    • 处理发送和接收端的传输

  2. 数据链路层

    • 将数据整理成帧
    • 提供验错和流控功能

  3. 网络层

    • 处理数据包在网络中的路由

  4. 传输层

    • 提供实际的通信功能
    • 主要协议：
      • TCP：面向连接、可靠、面向流
      • UDP：不可靠(尽力而为)的数据报文通信
      • QUIC：基于UDP的新型传输层协议

  5. 中间件层

    • 提供通用服务和协议
    • 功能：
      • 通信协议
      • 数据封装/解封装
      • 命名协议
      • 安全协议
      • 扩展机制(复制和缓存)

2. 通信类型

2.1 基本分类

  1. 瞬时通信 vs 持久通信

    • 瞬时：消息无法传递时直接丢弃
    • 持久：消息存储在服务器直到传递成功

  2. 同步通信 vs 异步通信

    • 同步点：
      • 请求提交时
      • 请求传输时
      • 请求处理完后

2.2 客户端/服务器通信特点

  1. 同步通信缺点：
    • 双方必须同时活跃
    • 客户端请求后被阻塞
    • 必须立即处理失效
    • 不适用于某些应用(如邮件系统)

3. 远程过程调用(RPC)

3.1 基本原理

  • 隐藏调用者和被调用者之间的通信
  • 通过过程调用机制实现分布式通信

3.2 参数传递注意事项

  1. 客户端和服务器可能有不同的数据表示
  2. 需要一致的编码机制：
    • 基本数据类型表示
    • 复杂数据类型表示

3.3 RPC类型

  1. 同步RPC

    • 遵循请求-响应模式
    • 客户端等待服务器返回

  2. 异步RPC

    • 客户端无需等待响应
    • 可继续执行其他操作

  3. 多RPC调用

    • 向多个服务器发送请求

3.4 现代RPC实现：gRPC & Protobuf

特点：

  1. 语言无关、平台无关
  2. 高效(比XML更小、更快)
  3. 扩展性和兼容性好
  4. 支持流式通信
  5. 严格的接口约束

4.1 Berkeley套接字

主要原语：

4.2 ZeroMQ高级消息模式

  1. Request-Reply模式
  2. Publish-Subscribe模式
  3. Pipeline模式

4.3 消息队列系统

特点：

  1. 支持异步持久通信
  2. 通过队列管理器管理消息
  3. 提供消息路由功能

5. Kafka消息系统

主要组件：

  1. Producer - 消息生产者
  2. Consumer - 消息消费者
  3. Broker - 消息处理节点
  4. Topic - 消息主题
  5. Partition - 主题分区

6. 多播通信

6.1 应用层多播

组织方式：

  1. 树状结构

    • 节点间唯一路径
    • 简单但可靠性较低

  2. 网络结构

    • 节点有多个邻居
    • 健壮性高

6.2 Gossip协议(流行病协议)

传播模型：

  1. Anti-entropy(反熵)模型

    • 最终一致性
    • 节点状态最终达成一致

  2. 流言传播模型

    • 具有良好扩展性
    • 适合大规模系统

这些通信机制为分布式系统提供了不同层次和类型的通信支持，系统设计者需要根据具体需求选择合适的通信方式。

分布式系统命名系统总结

1. 基本概念

1.1 名称

  • 由字符组成的字符串，用于标识分布式系统中的实体对象
  • 可标识的实体包括：
    • 硬件资源：主机、打印机、磁盘、文件等
    • 抽象资源：用户、邮箱、消息等

1.2 访问点(Access Point)

  • 对实体进行操作的接入点
  • 访问点的名称称为地址
  • 一个实体可以拥有多个访问点

1.3 标识符(Identifiers)

特性：

  1. 一个标识符至多引用一个实体
  2. 每个实体最多由一个标识符引用
  3. 标识符始终引用同一个实体(永不重用)

2. 无层次结构命名

2.1 基于广播/多播的命名解析

  1. 广播方式

    • 广播ID请求实体返回当前地址
    • 局限性：
      • 难以跨越局域网
      • 需要所有进程监听请求

  2. 多播方式

    • 适用于大规模网络
    • 降低带宽浪费

2.2 基于转发指针的命名解析

原理：

  • 实体移动时在当前位置留下指向新位置的指针
  • 客户端沿指针链搜索

问题：

  • 长传播链容易断开
  • 实体定位开销大
  • 地理可扩展性差

2.3 分布式散列表(DHT)

以Chord为例：

结构特点：

  1. 节点组织成环形结构
  2. 每个节点有m位标识符
  3. 实体被赋予唯一的m位健值
  4. 健值k的实体存储在满足id >= k的最小标识符节点上

Chord指状表特点：

  • 每个节点维护m个指针
  • 第i个指针指向当前节点后的2^(i-1)个位置
  • 支持对数级别的查找效率

3. 结构化命名

3.1 命名空间

基于命名图(naming graph)组织：

  • 叶节点：表示实体，存储实体属性、状态等信息
  • 目录节点：包含指向其他节点的边

3.2 名称解析机制

  1. 闭包机制
    启动名称解析的方式：
  • www.domain.com → 从DNS服务器开始
  • /home/user/file → 从本地NFS服务器开始
  • IP地址 → 直接路由到特定地址
  1. 链接类型
  • 硬链接：直接在命名图中按路径搜索
  • 软链接：节点存储另一个节点的绝对路径名

3.3 命名空间实现的三层结构

  1. 全局层

    • 包含根节点及近根节点
    • 特点：稳定，很少改变
    • 代表组织或组织群

  2. 行政层

    • 单个组织内管理的目录节点
    • 相对稳定，变化比全局层频繁
    • 代表同一组织的实体组

  3. 管理层

    • 频繁改变的节点
    • 代表本地主机、文件系统等
    • 由终端用户维护

4. 名称解析方式

4.1 迭代式解析

  • 客户端主动迭代查询每个解析服务器
  • 每个服务器返回下一步查询的服务器信息

4.2 递归式解析

  • 客户端发出一次请求
  • 服务器递归查询其他服务器
  • 返回最终结果

5. 基于属性的命名

5.1 LDAP(轻量级目录访问协议)

特点：

  • 结合数据库实现目录服务
  • 每个目录项包含(属性,值)对
  • 赋予唯一名字便于查找
  • 支持基于属性的查询

5.2 扩展性考虑

  1. 规模扩展性

    • 服务器需处理大量请求
    • 高层服务器面临更大挑战

  2. 地理扩展性

    • 通过复制提高可用性
    • 就近服务原则

分布式系统一致性与复制总结

1. 基础概念

1.1 复制的原因

  1. 性能提升
  2. 提高可用性

1.2 主要挑战

  1. 一致性保证
    • 确保冲突操作在任何地方按相同顺序执行
  2. 冲突类型
    • 读写冲突：读操作和写操作并发
    • 写写冲突：两个写操作并发

2. 数据中心一致性模型

2.1 连续一致性模型

定义三种偏差类型：

  1. 数值偏差

    • 不同副本间的数值可能存在差异

  2. 新旧偏差

    • 不同副本间数据新旧程度不同

  3. 顺序偏差

    • 不同副本更新操作的顺序和数量可能不同

2.2 Conit (一致性单元)

2.3 顺序一致性

定义：所有进程的操作按某种顺序执行，每个进程的操作按程序顺序出现。

2.4 因果一致性

定义：具有潜在因果关系的写操作必须以相同顺序被所有进程看到。

特点：

  • 弱于顺序一致性
  • 并发写操作可以以不同顺序被看到
  • 保持写操作的因果关系

3. 客户端中心一致性模型

3.1 单调读一致性

定义：如果进程读取数据项x的值，后续读取操作得到的值不会更旧。

3.2 单调写一致性

定义：一个进程的写操作必须在该进程的任何后续写操作前完成。

3.3 读写一致性

定义：进程的写操作结果必须对该进程的后续读操作可见。
